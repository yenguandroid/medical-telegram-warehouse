# Star Schema Design — Medical Telegram Warehouse

## Overview

The data warehouse uses a **star schema** — a central fact table surrounded by
denormalized dimension tables. This design was chosen over a snowflake schema
because our dataset is moderate in size, our analytical queries are
straightforward aggregations, and star schemas offer simpler SQL and faster
query performance without the need for complex joins.

## Design Decisions

### 1. Grain of the Fact Table
The grain of `fct_messages` is **one row per unique Telegram message per channel**.
A composite surrogate key (`message_id` + `channel_name`) handles the fact that
Telegram message IDs are only unique within a channel, not globally.

### 2. Surrogate Keys
All dimension tables use **MD5-based surrogate keys** (generated by `dbt_utils.generate_surrogate_key`).
This decouples the warehouse from Telegram's internal IDs and makes joins stable
even if source IDs change.

### 3. dim_channels — Pre-aggregated Stats
Channel-level aggregates (`avg_views`, `total_posts`, `total_images`) are stored
directly in `dim_channels` rather than computed at query time. This is a deliberate
denormalization that avoids expensive GROUP BY operations on every dashboard query.
The tradeoff is that these values are only as fresh as the last `dbt run`.

### 4. dim_dates — Full Calendar Spine
`dim_dates` uses `dbt_utils.date_spine` to generate a **complete, contiguous date
range** rather than pulling distinct dates from the fact table. This ensures that
dates with zero messages still appear in time-series charts, preventing misleading
gaps in visualizations.

### 5. Channel Classification
The `channel_type` field (Pharmaceutical / Cosmetics / Medical) is derived in
`stg_telegram_messages` using a CASE statement on known channel names. This
classification is intentionally kept in the staging layer so it flows consistently
into both `dim_channels` (as an attribute) and `fct_messages` (via the dimension join).

### 6. Materialization Strategy
| Layer    | Materialization | Reason                                              |
|----------|-----------------|-----------------------------------------------------|
| Staging  | View            | Always reflects latest raw data; low storage cost  |
| Marts    | Table           | Fast query performance for BI tools and dashboards |

### 7. Data Quality Gates
Three layers of data quality protection:
- **Staging filters**: future-dated messages, negative views, and null IDs are
  excluded before data reaches the marts.
- **dbt schema tests**: `unique`, `not_null`, `accepted_values`, and `relationships`
  tests run on every `dbt test` invocation.
- **Custom SQL tests**: `assert_no_future_messages`, `assert_positive_views`, and
  `assert_fct_no_orphan_channels` enforce business rules that generic tests cannot express.
